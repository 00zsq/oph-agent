# Oph Agent 下一步计划（缺口导向）

本文档只描述两件事：

1. 目前还缺什么。
2. 下一步按什么顺序推进。

## 1. 当前基线（已具备）

1. LLM 调用已稳定：Qwen/OpenAI 兼容模式。
2. 联网检索已可控：前端开关 + 后端参数透传。
3. RAG 已落地：静态知识库 + 检索服务解耦 + 引用补齐。
4. Function-call 已落地：按钮触发业务工具、普通输入默认禁用。
5. 医疗规范外置：系统提示词文件化加载。
6. 批量评测可执行：命中率与引用完整率统计。

## 2. 缺失能力清单（重点）

### P0（必须补齐）

1. 检索质量不足

- 现状：关键词检索为主，复杂问答命中率偏低。
- 影响：回答稳定性不够，易出现“未命中”或引用不充分。

2. 规则硬约束缺失

- 现状：医疗安全规则主要依赖提示词，缺少程序化审计。
- 影响：高风险场景一致性不可保证。

3. 可观测性不足

- 现状：前端看不到“是否联网/是否调用工具/调用了什么”。
- 影响：问题排查与验收成本高。

### P1（应尽快补齐）

1. 业务工具覆盖不足

- 现状：仅患者列表、诊断记录、预约管理 3 类。
- 影响：实际临床业务覆盖面不足。

2. 评测集规模不足

- 现状：样本数量较少，场景分层不够。
- 影响：优化方向不够准确，迭代收益难量化。

### P2（增强项）

1. 会话记忆与缓存策略不完善。
2. 追踪体系未接入（如统一 traceId、链路时延统计）。
3. 自动化回归测试覆盖不足。

## 3. 下一步执行计划

## 阶段 A（本周）：先提升准确率与可解释性

1. 检索升级到混合检索（关键词 + 向量 + 重排）。
2. 保持现有工具接口不变，只替换检索内部实现。
3. 输出结构化来源信息（内部知识/联网/业务工具）。

验收标准：

1. 内部问答命中率达到目标阈值（建议 >= 80%）。
2. 引用完整率达到目标阈值（建议 >= 95%）。
3. 回答中可明确区分来源类型。

## 阶段 B（下周）：扩展业务能力

1. 新增 2-4 个高频 Java 工具（按单文件拆分）。
2. 统一工具返回结构（分页、状态码、错误语义）。
3. 前端增加对应快捷按钮与提示文案。

验收标准：

1. 新增工具可稳定调用，失败可读、可追踪。
2. 按钮与工具 action 一一映射，且手动输入仍不触发业务工具。

## 阶段 C（第二周）：完善评测与上线前质量门禁

1. 扩充评测集到至少 60 条，按场景分桶：

- 普通医学问答
- 业务工具调用
- 时效性问题（联网）

2. 增加失败样本自动归档与回归复测。
3. 增加规则校验脚本（禁用词、风险提示完整性、引用格式）。

验收标准：

1. 三类场景均有独立统计结果。
2. 回归后指标无明显倒退。

## 4. 实施顺序建议

1. 先做阶段 A（准确率）
2. 再做阶段 B（能力覆盖）
3. 最后做阶段 C（稳定上线）

原因：先把回答质量和可解释性做好，再扩功能，整体返工最少。

## 5. 需要补充的外部输入

1. Java 新接口文档（路径、参数、响应、鉴权、错误码）。
2. 更多高质量知识文档（建议新增 30-50 篇）。
3. 可脱敏评测样本（建议 60 条以上，含期望要点）。
4. 医疗安全合规红线清单（必须输出项、禁止表达项）。

## 6. 下一个可直接开工项

建议立即开始：阶段 A 的“混合检索替换”。

完成定义：

1. 不改前端接口。
2. 不改业务工具接口。
3. RAG 命中率和引用完整率有可量化提升。
